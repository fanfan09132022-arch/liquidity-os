## 一、项目定义

LiquidityOS 是一个面向个人使用的交易决策工作台，用于将宏观流动性分析、板块热度判断、Meme 资产观察、每日记录与历史回看整合进同一套工作流中。

当前产品不是公开 SaaS，也不是多用户系统。  
当前阶段定位为：

- 个人日常交易工作台
- 交易框架沉淀工具
- 内容创作与公开输出的底层支撑工具

---

## 二、项目目标

LiquidityOS 服务于两个长期目标：

1. 建立一套可重复执行的 Meme 币研判流程
2. 将交易框架、观察方法和决策过程沉淀为可复盘系统

## 三、当前产品状态

### 3.1 当前部署形态

- 前端：Cloudflare Pages
- 数据接口：Cloudflare Worker
- 本地存储：浏览器 `localStorage`

### 3.2 当前已打通的主链路

以下链路已通过真实运行验证：

- 更新数据 / 强制更新 → Worker 返回 → 页面写入当天快照
- 缓存命中 → 当天快照回填
- 历史日期切换 → 只读历史快照 → 切回今天恢复当日数据
- `/api/fred` → `/api/all` → L1 自动区展示
- Top 50 Meme 榜单 → Worker 返回 → 页面真实渲染
- Alpha Scanner 第一轮自动支撑数据 → 部分接通，且不污染手动判断

### 3.3 当前阶段判断

当前产品已完成从原型到独立站 MVP 的迁移，核心闭环成立，后续重点为：

- 页面级 UI / 交互重构
- 数据能力继续补强
- 文档与实现口径统一

## 四、核心框架：L0-L4

LiquidityOS 的核心逻辑是从宏观到微观逐层过滤，上层约束下层。

```
L0 → 确定仓位上限和操作偏见（进攻 / 防守）
  ↓
L1 → 确认总流动性方向
  ↓
L2 → 确认稳定币弹药与链上战场
  ↓
L3 → 确认 Meme 板块整体热度
  ↓
L4 → 观察存量结构与增量机会，形成最终交易判断
```

**核心原则：**

- 上层信号决定下层操作空间
- 下层机会判断不能脱离上层环境独立使用
- Hero 为综合信号，不等同于仓位指令
- L0 / L1 为仓位约束层，不直接替代选币逻辑

## 五、产品结构

当前页面按产品逻辑重构为四层：

### 5.1 主控层

用于回答：

- 今天是什么状态
- 数据是否是新的
- 仓位偏向是什么
- BTC 当前处于什么位置

包含：

- 顶部操作区
- 当前数据来源 / 更新时间状态条
- Hero 主控卡
- L0-A + L0-B 仓位依据区

### 5.2 决策层

用于承载宏观到中观判断依据。

包含：

- L1 · 净流动性
- L2 · 稳定币弹药
- L3 · Meme 板块
- F&G
- BTC 位置辅助

该层正在向统一决策带演进，并补充轻趋势图表达。

### 5.3 执行工作台

用于从判断落到具体标的与操作对象。

包含：

- 自动市场面板：Top 50 Meme 榜单
- 主工作区：存量观测站
- 增量机会区：Alpha Scanner

### 5.4 记录与诊断层

用于复盘、记录与排错。

包含：

- 历史回看
- 近 30 天摘要
- 笔记
- 系统状态区

## 六、功能模块定义

### 6.1 Hero 主控卡

**作用：** Hero 为综合信号，用于快速回答当前环境偏强还是偏弱。

**当前输入：**

- L2
- L3
- F&G
- L4

**当前边界：**

- L0 / L1 不直接并入 Hero 平均分
- L0 / L1 独立承担仓位约束职责

**当前输出结构：**

- color
- score
- reason

### 6.2 L0：仓位约束层

#### L0-A：全球流动性周期

**当前状态：**

- 扩张
- 过渡
- 收缩

**当前语义：**

- 扩张：偏进攻
- 过渡：控制仓位
- 收缩：偏防守

**当前实现：**

- 手动选择
- 已补充简短解释文案
- 已进入主控层与 Hero 同层展示

#### L0-B：BTC 周期位置

**当前承载字段：**

- BTC 现价
- 24h 变化
- 200MA
- 相对 200MA 偏离
- MVRV Z-Score

**当前实现状态：**

- BTC 现价 / 200MA / 偏离：自动展示
- MVRV Z-Score：手动填写位

**MVRV 当前保留：**

- 数值输入
- 数据日期
- 来源说明
- 外链入口

### 6.3 L1：全球净流动性

**公式：** GNL = Fed − TGA − RRP

**当前自动数据：**

- Fed
- TGA
- RRP
- GNL
- 数据日期
- 来源
- 更新时间

**当前产品逻辑：**

- 自动优先（FRED）
- 手动兜底
- 自动区默认显示
- 手动区仅在自动数据缺失时承担兜底职责

### 6.4 L2：稳定币弹药

**作用：** 判断场内总弹药是否增强，热钱当前集中在哪条链。

**当前状态：**

- 已有统一信号结构
- 已进入决策带
- 已补充轻趋势图表达

### 6.5 L3：Meme 板块热度

**作用：** 判断 Meme 板块整体热度，热度是否扩散或减弱。

**当前状态：**

- 已有统一信号结构
- 已进入决策带
- 已补充轻趋势图表达

### 6.6 F&G：情绪层

**作用：** 作为市场情绪辅助信号，补充环境判断。

**当前状态：**

- 已进入决策带
- 已补充轻趋势图表达

### 6.7 L4：存量与增量结构

L4 当前已明确拆成三种不同职责。

#### A. Top 50 Meme 榜单

**职责：** 自动市场观察，观察市场全貌与主要 Meme 资产分布。

**当前状态：**

- 数据源：CoinMarketCap
- 当前仅做展示
- 不并入评分逻辑
- 当前为辅助观察区

#### B. 存量观测站

**新定位：** 用于跟踪近期看好、准备建仓，或已经有仓位的币，是主观重点跟踪的观察台。

**当前字段：**

| 字段 | 说明 |
|------|------|
| Token | 代币名称 |
| 状态 | 观察中 / 准备建仓 / 已有仓位 |
| MCap | 市值 |
| 24h | 24 小时涨跌 |
| 7d | 7 日涨跌 |
| 1m | 1 个月涨跌 |
| V/MC | 换手活跃度 |
| 筹码集中度评分 | 健康 / 中性 / 集中 |
| 池子深度比 | 深 / 中 / 浅 |
| 备注 | 文本输入 |

**当前输入方式：**

- 状态：分段选择
- 筹码集中度评分：健康 / 中性 / 集中
- 池子深度比：深 / 中 / 浅
- 其余：数值输入或文本输入

#### C. Alpha Scanner

**定位：** 增量新机会筛选区。

**当前结构：**

- 标准化输入
- 自动支撑数据
- 手动判断区

**第一轮自动拉数边界：**

- 支持链：solana、bsc
- 输入主键：chain + token address
- 自动值只做支撑，不覆盖手动判断

**当前自动化状态：**

| 维度 | 状态 |
|------|------|
| momentum | 部分自动化已打通 |
| chips | BSC 当前为 not_supported |
| pool | 自动值第一轮暂缺，保留手动输入 |

## 七、数据架构

### 7.1 当前主要数据源

| 数据源 | 用途 |
|--------|------|
| FRED | Fed / TGA / RRP / GNL |
| Binance | BTC 现价 / 历史价格 / 200MA |
| CoinMarketCap | Top 50 Meme 榜单 |
| Birdeye | Alpha Scanner token 维度支撑数据 |
| GeckoTerminal | Alpha Scanner pool 数据（第一轮未完全打通） |
| Glassnode | MVRV 后续自动化规划 |

### 7.2 当前前后端职责边界

**Cloudflare Worker 负责：**

- 外部数据请求
- 数据清洗与统一返回结构
- 聚合接口输出

**前端负责：**

- 页面结构与交互
- 本地快照和历史读取
- 手动输入与展示逻辑
- 轻趋势图渲染（当前使用本地快照）

## 八、快照与历史能力

**当前已实现：**

- 每日快照保存
- 按日期读取
- 近 30 天摘要
- dirty-save 保存边界
- 历史只读查看

**当前原则：**

- 历史查看不应误触发保存
- 缓存回填不应污染手动状态
- 更新成功后的最新数据不应被旧状态覆盖
