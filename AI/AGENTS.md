## 这份文件的用途

这是当前项目给 Codex 的 **执行入口规则**。

它的目标不是让 Codex 重新理解整个项目，  
而是让 Codex 在进入项目后，按统一方式执行：

- 先确认当前任务包是否完整
- 再在受控范围内实现
- 再做最小必要验证
- 最后输出结构化执行交接单

这份文件不是完整知识库。  
它是 Codex 的执行规则入口。

项目背景、架构现实、流程路由、skills 都在 `AI/` 目录下。

## 四、执行总原则

### 1. 包外不做

你只在当前实现包边界内工作。

不要：

- 顺手修别的东西
- 顺手重构
- 顺手改 unrelated 区域
- 看到旧问题就一起修
- 自动把本轮变成更大任务

### 2. 最小改动优先

优先选择：

- 最少文件
- 最小改动面
- 最低副作用
- 最易 review 的实现方式

### 3. 简洁优先

优先写：

- 简单
- 清楚
- 可维护
- 与现有结构一致

不要为了“更优雅”而过度重构。  
也不要为了“快”写明显 hack 的补丁，除非实现包本身就是临时修复。

### 4. 根因优先

如果这是 bug fix 或 correctness fix：

- 优先基于根因修改
- 不要只做表面 symptom patch
- 如果根因仍然不明，先停下说明

### 5. 验证优先

代码改完不算完成。  
至少做与当前包相匹配的最小必要验证。

## 六、收到实现包后的执行步骤

### 第一步：确认包是否合格

开始前检查：

- 任务是不是只有一个主目标
- 允许修改文件是否明确
- 禁止修改文件是否明确
- 验证步骤是否明确
- 延后事项是否明确

如果不合格，停下并说明缺什么。

---

### 第二步：先做最小实现路径判断

在真正改代码前，先判断：

- 能否只改允许文件中的最低风险文件
- 能否避免动高风险文件
- 能否避免影响现有 working runtime
- 能否不把 detail 问题扩散到 homepage 结构
- 能否不碰 Worker

如果能，优先选最小路径。

---

### 第三步：严格按包实现

执行过程中：

- 只做当前包要求的事
- 不扩大 scope
- 不主动增加第二目标
- 若发现问题已超出当前包，停止扩展，并在回传中说明

---

### 第四步：做最小必要验证

完成实现后，至少检查：

- 当前目标对应的用户可见变化是否成立
- 是否破坏相关页面 / 入口 / wiring
- 是否有明显 console / runtime 错误
- 是否与实现包的验证步骤一致

如果无法验证，要明确写出：

- 哪些没验证
- 为什么没验证
- 当前不确定项是什么

---

### 第五步：输出执行交接单

完成后，不要输出长篇过程回放。  
请按以下文件要求输出：

- `AI/03-skills/custom/01-执行结果汇报官.md`

如果本轮输入来自 handoff-safe 或 compact 后上下文，
则回传必须继续保持紧凑：

- 优先使用最小字段
- 不解释实现过程
- 不复述背景
- 不展开验证细节
- 不重复 package 内容

如需完整信息，写入结果文件，而不是写入聊天。

## 七、你应该如何处理不确定性

### 情况 1：需求边界不清

不要自行定义产品方向。  
停下并说明：

- 哪些信息不足
- 为什么当前无法安全实现
- 最需要补充的最小信息是什么

### 情况 2：发现必须越界

如果你发现为了完成当前目标，必须修改未授权文件：

- 不要直接越界改
- 先停下
- 说明为什么当前包不足
- 说明最可能需要新增哪些文件授权

### 情况 3：发现问题根因不明

如果是 bug / regression，但你发现根因并不明确：

- 不要强行 patch
- 说明当前怀疑层
- 说明已有证据
- 说明还需什么信息

### 情况 4：发现更大的潜在重构机会

不要自动展开。  
即使你看到了“更理想方案”，也应：

- 先完成当前包
- 把更大方案写进延后事项或风险说明
- 不要擅自替换当前包目标
